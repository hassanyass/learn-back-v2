# The Feedback Flow: End-to-End Architecture

This document explains the technical architecture and end-to-end data flow of the LearnBack Feedback System. It triggers when a session concludes and the user requests their final performance report.

---

## 1. How the Feedback is Triggered

The process starts when the user finishes a session or clicks "End Session". The frontend navigates to the Feedback page, which immediately makes an API call to the backend:

**Endpoint:** `GET /session/{session_id}/feedback`

This endpoint uses JWT bearer authentication to verify the user owns the session. If authorized, it passes control to the `FeedbackService`.

---

## 2. Caching & Concurrent Generation Locks

Before doing any heavy processing or LLM calls, the service checks for existing data to save money and time:

1. **Cache Check:** The system checks the `LearningSession.feedback_data` column in the database. If a report has already been generated (e.g., the user refreshed the page), it instantly returns the cached JSON.
2. **Generation Lock:** If the feedback is currently being generated by another simultaneous request, the `session_state` will have `"feedback_status": "generating"`. The system will wait (sleep) rather than kicking off a duplicate LLM call. If it's free, it acquires the lock.

---

## 3. Exactly What Data is Retrieved?

To generate a highly personalized and accurate report, the `FeedbackService` pulls three specific pieces of data from the database:

### A. The Syllabus (From `SlideDeck`)
It pulls the original `segmented_json` from the uploaded SlideDeck. This provides the structure: what topics were supposed to be covered, and what concepts were inside those topics.

### B. The State & BKT Scores (From `LearningSession.session_state`)
It pulls the deeply nested `session_state` JSON. This tells the system exactly where the user left off. Specifically, it retrieves:
* The `bkt_score` (mastery probability) for every single concept.
* Any `misconceptions` flagged by the Judge during the session.
* The `kido_memory` (the Mind Map summaries) for completed points.
* Attempt counts (how many tries it took to get something right).

### C. The Chat History (From `SessionMessage`)
It pulls every chat bubble sent during the session, ordered chronologically. It formats this into a giant transcript:
```text
[KIDO]: What is a neural network?
[USER]: It's a system inspired by the brain...
[KIDO]: That's cool! Can you explain...
```

---

## 4. The LLM Feedback Generation

Once the data is retrieved, the service checks if the user actually sent any messages. If the history is empty, it skips the LLM entirely (to save API costs) and uses a fallback.

If there is a conversation history, the data is bundled into a prompt:
* **The Prompt:** Contains the `FEEDBACK_SYSTEM_PROMPT`, a list of the concepts alongside their BKT scores, and the raw conversation transcript.
* **The Call:** The prompt is sent to the primary LLM (Groq, typically `llama-3.1-8b-instant`).
* **Timeout & Fallback:** The LLM call has a strict **10-second timeout**. If Groq takes too long, or if the API is down, the system catches the error and gracefully switches to a "Deterministic Fallback" which generates a basic report based purely on math (without LLM-generated text).

**What the LLM Returns:**
The LLM is strictly instructed to return a JSON object containing `topic_cards`—an array where each topic gets a specific list of `strengths`, `weaknesses`, and `suggestions` based on reading the chat transcript.

---

## 5. Data Assembly & Enrichment

The LLM text is great, but the frontend needs hard math to render the UI (like progress bars). The service now stitches the LLM's text output together with the BKT math:

1. **Topic BKT Calculation:** It averages the BKT scores of all points inside a topic to get the Topic BKT.
2. **Understanding Levels:** It assigns a label based on the math:
   * `strong` (Avg BKT ≥ 0.7)
   * `good` (Avg BKT ≥ 0.4)
   * `weak` (Avg BKT < 0.4)
   * `skipped`
3. **Point Breakdown:** It maps every single concept, attaching its exact BKT score, total attempts, and any specific misconceptions that occurred.
4. **Overall Mastery:** It averages all BKT scores across the entire session to create a single `overall_mastery` percentage (e.g., 82%).

---

## 6. Persistence & The Final Response

Finally, the giant assembled JSON object is saved to the database:
* `session.feedback_data = final_feedback_data`
* The lock is released (`"feedback_status": "ready"`).

The backend then returns this rich JSON object to the frontend.

**What the Frontend Gets (The Final JSON Structure):**
```json
{
  "session_id": 123,
  "overall_mastery": 82,
  "duration_minutes": 15,
  "strengths": ["Clear explanation of core concepts", "Good use of real-world examples"],
  "weak_areas": ["Confused terminology in Topic 2"],
  "topics": [
    {
      "id": "topic-0",
      "title": "Introduction to Neural Networks",
      "status": "completed",
      "understanding": "strong",
      "bkt_score": 0.85,
      "feedback": "Clear explanation of core concepts. However, confused terminology...",
      "misconceptions": ["Mixed up weights and biases"],
      "points": [
        {
          "title": "What is a Neuron?",
          "status": "completed",
          "bkt_score": 0.90,
          "attempts": 1,
          "misconceptions": []
        }
      ]
    }
  ]
}
```

The frontend JavaScript takes this JSON and uses it to render the beautiful progress bars, the feedback cards, and the global strengths/weaknesses on the feedback dashboard.
