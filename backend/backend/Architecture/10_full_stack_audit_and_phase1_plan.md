- `backend/backend/main.py`

    - `backend/backend/routes/dashboard_router.py` (`/dashboard`)

Main service modules:

- `backend/backend/services/session_service.py` (session orchestrator for chat/widget/mindmap/hints)

## B) Issues Found (Broken / Missing Integrations)

### Auth flow mismatches

1) Frontend expects user details from `/auth/login` that backend does not return.
   - Frontend: `frontend/auth.js` stores `data.user_id`, `data.username`.
   - Backend: `backend/backend/routes/auth_router.py` returns `TokenResponse` only (`access_token`, `token_type`).

2) Frontend dashboard calls `GET /auth/me`, but backend has no such endpoint.
   - Frontend: `frontend/dashboard.js` → `window.LearnBackAPI.request('/auth/me')`
   - Backend: no `/auth/me` route is present.

### Dashboard flow mismatch

Backend `GET /dashboard` returns a `DashboardResponse` aggregate-only payload:
- No `remote.user`
- No `remote.sessions[]`

Frontend `frontend/dashboard.js` only accepts a remote payload if it contains `remote.user` and `remote.sessions`.
Result: Dashboard usually falls back to `SessionStore` and becomes localStorage-driven, not backend-driven.

### Upload + slide parsing flow breaks (contract mismatch)

Backend ingestion endpoint:
- `POST /ingestion/upload-slides` returns `{ pdf_storage_url, segmentation }` (and persists `SlideDeck`).
- It does **not** return `document_id` or `slide_deck_id`.

Frontend upload flow:
- `frontend/upload_slides.js` requires `uploadResponse.document_id || uploadResponse.id` and throws if missing.
Result: upload → session init does not proceed with current backend response.

### Session bootstrap mismatch (session_state is dropped client-side)

Backend session bootstrap:
- `GET /session/{session_id}` returns `session_state` plus topic titles and metadata.

Frontend normalization:
- `frontend/apiClient.js` `normalizeSessionBootstrap()` does not preserve `session_state`,
  so `frontend/js/core/SessionState.js` often starts without nested topics/points.

### Slides/PDF view not connected to ingestion

Frontend slide pane uses localStorage/hardcoded fallback:
- `frontend/sessionSplitPane.js` loads `localStorage.getItem('learnback_pdf_url') || 'sample.pdf'`

Backend provides `SlideDeck.pdf_storage_url`, but it is not returned in session bootstrap, and upload flow doesn’t store it.
Result: session frequently shows `sample.pdf` instead of the uploaded deck.

### WebSocket security is missing (session hijack risk)

Backend WS:
- `backend/backend/routes/session_router.py` WS endpoint accepts connections without JWT verification.
Result: anyone with a `session_id` can connect and mutate/persist conversation for that session.

### Ownership checks missing on hint/feedback endpoints

Hint:
- `POST /session/{id}/hint` uses JWT auth but does not verify `LearningSession.user_id == user_id` before acting.

Feedback:
- `GET /session/{id}/feedback` uses JWT auth but does not verify ownership before returning/generating feedback.

### Feedback flow mismatches

Backend route exists:
- `GET /session/{session_id}/feedback`

Frontend API client calls non-existent endpoints:
- `frontend/apiClient.js` tries `/api/session/{id}/feedback` then `/api/feedback/session/{id}`

Query param mismatch:
- Session page sends `feedback.html?sessionId=...` (in `frontend/session.js`)
- Feedback page reads `?id=...` (in `frontend/feedback.js`)

### Hardcoded / placeholder logic

- `backend/backend/services/document_service.py` storage upload is a stub (returns example URL).
- PPTX upload sets `pdf_storage_url` to a `placeholder://...` scheme.
- Frontend production API base URL is a placeholder for non-local hosts (`frontend/apiClient.js`).
- Asset casing issues likely break on case-sensitive hosting:
  - `frontend/session.html` references `Logo/...` but folder is `frontend/logo/`
  - `frontend/session.html` references `Idle.lottie` but file is `frontend/idle.lottie`

### Unconnected / inconsistent modules

- `backend/backend/services/evaluator_service.py` exists but the main session chat pipeline in `session_service.py`
  calls an internal `_call_evaluator()` instead of using `EvaluatorService.evaluate_message()`.
- `backend/backend/services/session_service.py` duplicates LLM fallback logic instead of using `core/llm_manager.py`.
- `backend/backend/services/anti_cheat_service.py` exists but is not used.
- Frontend widget engine modules exist (`WidgetFactory.js`, `WidgetDispatcher.js`) but session UI uses a stub textarea modal.

### Security hygiene note

The backend `.env` contains secrets/credentials and should not be committed in production workflows.
(**Do not** paste secret values into docs or issues; rotate keys if they were exposed.)

## C) Session Flow Breakdown

### Intended flow (based on implemented backend + `Architecture/04_session_orchestrator.md`)

1) User logs in → frontend stores JWT.
2) User uploads slides (PDF/PPTX) → backend extracts raw text and segments into <= 4 topics → persists `SlideDeck`.
3) User starts a session → backend builds full nested `session_state` (topics + points) from `SlideDeck.segmented_json` and persists `LearningSession`.
4) Session page bootstraps via REST (`GET /session/{id}`) → frontend opens WS (`/ws/session/{id}`).
5) Chat runs over WS typed payloads: `chat`, `mind_map_submit`, `widget_submit`.
6) Backend persists messages and mutates `session_state` through the Evaluator → BKT → Kido pipeline.

### Actual flow (current repo)

- Break #1: upload → session init fails in `frontend/upload_slides.js` because ingestion response lacks `document_id`.
- Break #2: session REST bootstrap partially fails on frontend because `session_state` is dropped in `frontend/apiClient.js`.
- Break #3: slides view uses `sample.pdf` due to missing wiring from backend `pdf_storage_url`.
- Break #4: feedback navigation uses mismatched query params and mismatched API endpoints.
- Security break: WS has no auth, and hint/feedback don’t enforce ownership.

## D) Step-by-Step Fix Plan (Phases)

> The project already has “Phase” docs in `backend/backend/Architecture/`.
> The plan below maps fixes to concrete files and dependencies.

### Phase 1 — Backend fixes (session-related core logic only)

**Goal:** Make sessions bind to the correct slide deck and close security holes in session-related endpoints.

1) Bind `LearningSession` to a `SlideDeck`:
   - Add `LearningSession.slide_deck_id` FK → `slide_decks.id` (nullable for legacy rows).
   - Update `POST /session/create` to store `slide_deck_id` of the chosen deck.
   - Update `SessionService` and `FeedbackService` to load segments from that deck (fallback to latest deck if null).

2) Ownership enforcement:
   - Hint endpoint must verify session ownership before generating hints.
   - Feedback endpoint must verify session ownership before returning/generating reports.

3) WebSocket authentication:
   - Require JWT for `/ws/session/{session_id}` (token via query param is simplest with browser WS).
   - Verify ownership before accepting the WS connection.

**Files (expected):**
- `backend/backend/models/core.py`
- `backend/alembic/versions/*.py` (new migration)
- `backend/backend/routes/session_router.py`
- `backend/backend/services/session_service.py`
- `backend/backend/routes/feedback_router.py`
- `backend/backend/services/feedback_service.py`

**Dependencies:** Frontend WS URL must be updated later to pass the token.

### Phase 2 — API alignment (frontend ↔ backend)

- Preserve backend `session_state` in frontend normalization (`frontend/apiClient.js`).
- Point feedback client to `GET /session/{id}/feedback`.
- Align feedback query params (`?id` vs `?sessionId`).
- Update WS client to pass JWT token.
- Decide auth contract for user identity:
  - Add `/auth/me` OR stop frontend from expecting user info on login. (**UNKNOWN preferred direction**)

### Phase 3 — Frontend `session.html` hardening

- Fix asset casing (`Logo/` vs `logo/`, `Idle.lottie` vs `idle.lottie`).
- Stop defaulting slides to `sample.pdf`; render slides using backend-derived URL or show a graceful fallback.

### Phase 4 — Slide parsing + segmentation integration (Groq)

- Ingestion should return `slide_deck_id` (aka “document id”) so frontend can target the correct deck.
- Implement real storage upload (Supabase) or explicitly represent “no PDF available” for PPTX.
  - PPTX → PDF conversion availability is **UNKNOWN** for deployment environment.

### Phase 5 — Chat integration polish

- Standardize WS payload naming conventions (labels, widget types, widget data).
- Optionally replace widget stub textarea with `WidgetFactory/WidgetDispatcher` rendering.

### Phase 6 — Cleanup and state management

- Remove dead/duplicate code paths (LLM fallback duplication, unused services).
- Update E2E tests to match real endpoints and supported file types.
- Remove secrets from repo tracking and rotate exposed keys.
